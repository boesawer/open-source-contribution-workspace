# Issue Scorecard

I use this checklist to decide whether an issue is a good fit before opening a pull request.

Score each item from 0 to 2:

- 0 = unclear or risky
- 1 = possible, but needs more context
- 2 = clear and approachable

## Criteria

1. Clarity
   - Is the request concrete?
   - Are expected results, screenshots, or reproduction steps included?

2. Scope
   - Can the work be kept small?
   - Does it avoid broad refactors or new systems?

3. Repository Health
   - Is the repository active?
   - Do maintainers respond to issues or pull requests?

4. Setup
   - Are setup instructions available?
   - Are install and test commands clear?

5. Verifiability
   - Can the change be tested or checked?
   - Is there a relevant script, test, or manual verification path?

6. Availability
   - Is the issue unassigned?
   - Is there no open pull request already solving it?

7. Stack Fit
   - Does it match JavaScript, TypeScript, React, Node.js, or documentation work?

8. Contribution Value
   - Would the change be useful to the project?
   - Would it be understandable to review?

## Interpretation

- 13-16: strong candidate
- 9-12: possible candidate
- 5-8: likely skip
- 0-4: skip
